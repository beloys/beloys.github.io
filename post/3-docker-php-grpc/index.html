PHP gRPC client + server

May 30, 2020 • 3 min read

Benefits of gRPC in microservice architecture

1. Strict contract for API

Its one f the most required thing for me. API should strictly follow interface, no other way.

2. No more DTO needed

Protoc will generate all DTOs like clients and messages. No need more parse/map/hydrate responses.

3. Easy to communicate between microservices written in different languages.

As proto file its just a contract and implementation encapsulated, we can easily rewrite microservice from scratch, and we just should implement interface. GRPC will take care about transport layer and adapter for the most popular languages.

4. Speed

GRPC use HTTP 2.0 as transport for serialized data. The binary format and keep alive connections make gRPC really fast.

5. Streaming

Great feature, not so often required, but good to have it out of the box.

6. Easier to maintain with backward campatibility

Just need follow few rules in proto file editing.

Limitations for PHP

PHP dont have any native concurrency, so some gRPC features not possible to use. One of them - streaming from PHP server. Here you can see work in progress so mb in future this feature will be available. I suppose that possible implement it based on Swoole loop. For this moment I recommend for usage RoadRunner.

Things that gRPC makes more complicated

  • Load balancing (because of keep alive connections)
  • Http + JSON supported by 99.99% of languages. If you will wanna use any language that doesnt have gRPC library - you will got a big problem.

Here how looks proto file, its just an interface for your services

syntax = "proto3";

package grpc.health.v1;

service Health {
    rpc Check(HealthCheckRequest) returns (HealthCheckResponse);
}

message HealthCheckRequest {
    string service = 1;
}
message HealthCheckResponse {
    enum ServingStatus {
        UNKNOWN = 0;
        SERVING = 1;
        NOT_SERVING = 2;
        SERVICE_UNKNOWN = 3;
    }
    ServingStatus status = 1;
}

PHP Service

namespace App\Service;

use Grpc\Health\V1\HealthCheckRequest;
use Grpc\Health\V1\HealthCheckResponse;
use Grpc\Health\V1\HealthCheckResponse_ServingStatus;
use Grpc\Health\V1\HealthInterface;
use Spiral\GRPC;

class HealthService implements HealthInterface
{
    public function Check(GRPC\ContextInterface $ctx, HealthCheckRequest $in): HealthCheckResponse
    {
        return (new HealthCheckResponse())->setStatus(HealthCheckResponse_ServingStatus::SERVING);
    }
}

PHP Client

$client = new HealthClient('127.0.0.1:8080', [
    'credentials' => ChannelCredentials::createInsecure(),
]);

[$reply, $error] = $client->Check((new HealthCheckRequest())->setService('test'))->wait();
/** @var HealthCheckResponse|null $reply */

if (! $reply) {
    throw new Exception(sprintf('No reply %s', json_encode($error)));
}

echo $reply->serializeToJsonString();

My recommendations:

  • If you wanna speed up build time, store prebuilds in private Docker repository.
  • Don’t add to git any generated code, its just make a big mess in diff.
  • Store proto files in different composer repositories, write hook that will collect all of them and execute protoc.
  • If you use any queue (as RabbitMQ), you can store serialized data in message body and use the same DTO for sync and async communications.
