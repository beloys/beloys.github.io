7 tips how to make your PHP project better

May 2, 2020 â€¢ 4 min read

1. Dont use nullable signatures.

Every null should be checked and handled if it needed. It’s easy to miss it. If function more strict in parameters it creates less unexpected behavior and starts to be more predictable for a developer. Write in the strictest way when it possible and don’t use types only when non-strict solution has less codebase then strict in N times. For basic types use initial values as, 0.0/''. For objects use Null objects.

2. Collections instead of arrays.

In my practice i never seen that Collection has different type of objects. But array its mostly like a trash bin with paper, pencils, used condoms. Not the best experience to put your hand inside to find one that you need. PHP 7.4 has short arrow functions, so sort/filter/map prettier with collections.

$collection = new ArrayCollection([
    new FooClass(16),
    new FooClass(3),
]);
$collection->add(new FooClass(115));
$collection->add(new FooClass(1));

return $collection->map(fn(FooClass $el) => $el->getValue())
    ->filter(fn(int $el) => $el % 2)
    ->first();

3. PHPStan.

PHPStan - its static code analyser for PHP, with a lot of features such a Generics, Class-strings, etc.

class AnyEntity {
    /**
     * @var Collection<int,User> $users
     */
     private Collection $users;

    public function __construct(){
        $this->users->add(new User())
        $this->users->add(new Profile()) // -> error
    }
}

Catch function call on null

public function printUser(?User $user){
    if ($user){
        $this->printer->print(sprintf('1: %s', $user->getUuid()->toString())); // -> check passed
    }
    
    $this->printer->print(sprintf('2: %s', $user->getUuid()->toString())); // -> error
}

It’s easily use it from the beginning, but if project already started you can add check to new classes and add ignore rules by the folder, filename pattern or error pattern.

4. Codeception.

Codeception - it’s a bundle of tools for testing. All you need will be there. It has modular structure and easily to extend it. You can initialize codeception in every module of your system, and your tests will be very close to implementation. There will be possibility to start only module-depended tests and run all tests in project at once, need just add path where Codeception should search tests.

# -> Discovery codeception configs
# src/Modules/User/tests
# src/Modules/Setting/tests
# src/tests
include:
  - src/*
paths:
  tests: tests
  output: tests/_output
  data: tests/_data
  support: tests/_support
params:
  - .env.test
  - .env.test.local
coverage:
  enabled: true
  include:
    - src/*

5. CI/CD.

Tests without CI - useless. I don’t know other way how to keep tests updated. PHPStan without CI will be useless too. Any checks such as code coverage, name requirements, code style can be added to pipeline once time and pipeline will care about processes after every push. If you use Doctrine ORM, you can validate mapping with DB scheme in a pipeline. So nothing talk about there, just start to use it, keep your codebase quality level high and don’t spent time on routine deployment.

image: php:7.1.1
pipelines:
  default:
    - step:
        script:
          - apt-get update && apt-get install -y unzip
          - curl -sS https://getcomposer.org/installer | php -- --install-dir=/usr/local/bin --filename=composer
          - composer require phpunit/phpunit
          - vendor/bin/phpunit

6. Git hooks.

Everyone knows that git commit messages should be meaningful and short. But how check this commit doesn’t have known issues? If CI/CD processes helps keep clean code and find bugs after push, git hooks prevents getting in git history “broken” commits. One rule - hook should have minimal environment dependency. Run unit tests, static code analyzers, fix code styling (PHP CS Fixer), etc. Below a BrainMaestro/composer-git-hooks config example.

{
  "scripts": {
    "post-update-cmd": [
      "cghooks update"
    ],
    "post-install-cmd": [
      "cghooks add --ignore-lock"
    ]
  },
  "extra": {
    "hooks": {
      "pre-commit": [
        "echo committing as $(git config user.name)",
        "php-cs-fixer fix .",
        "phpstan analyse"
      ],
      "pre-push": [
        "php-cs-fixer fix --dry-run .",
        "codecept run unit",
        "phpstan analyse"
      ]
    }
  }
}

7. Use composer packages.

My opinion that an application should contain only business logic. Usually assist classes mark by @internal annotation, but I totally don’t like this way. Developer shouldn’t see any assist logic in the business logic layer. It’s should be hided like a framework classes in vendor folder and expose only an API. Don’t worry if packages will be a lot, keep naming standards, use prefixes and in future it will help new developers make learning curve shorter because on the front will be only matters.

{
  "name": "<your-name>/<your-project-name>",
  "require": {
    "php": "7.4.*",
    "<your-name>/component-<component-name>": "^1.3.0",
    "<your-name>/component-<component-name>": "^1.0.0",
    "<your-name>/lib-<library-name>": "^1.0.0",
    "<your-name>/lib-<library-name>": "^1.4.0",
    "<your-name>/module-<module-name>": "^1.4.0",
    "<your-name>/module-<module-name>": "^1.1.5",

  },
  "repositories": {
    "<your-name>/component-<component-name>": {
      "type": "vcs",
      "url": "git@gitlab.com:<your-name>/component-<component-name>.git"
    }
  }
}

If you have a lot of private repositories, you can use private packages or add tags to description in git store provider #package-composer. Next need only write a small tool for updating repository list in the package.json.
